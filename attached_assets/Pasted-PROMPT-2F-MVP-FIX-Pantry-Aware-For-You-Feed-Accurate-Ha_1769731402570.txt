DEFINITIONS
- Pantry items have a status/state: Have / Maybe / Need.
- A recipe contains a list of ingredients.
- The recipe card must show per-ingredient classification as Have/Maybe/Need (or summary counts if that’s what the UI supports; follow existing UI pattern).

WHAT YOU MUST IMPLEMENT
A) Pantry Lookup + Normalization Layer (deterministic)
1) Build a normalized pantry index:
   - Keyed by a normalized ingredient name (lowercase, trim, remove punctuation, collapse spaces).
   - Map to pantry status: HAVE | MAYBE | NEED (use the app’s current enum/strings; don’t invent new ones if they exist).
2) Build a normalized recipe ingredient list:
   - From FatSecret recipe payload (wherever your recipe card gets ingredients).
   - Normalize the ingredient display names using the same function.
3) Matching rule:
   - Exact normalized match first.
   - If your codebase already has synonym support or alias mapping, use it. If it does not, DO NOT invent a synonym system in this prompt.
   - If no match, default to NEED.

B) Per-Recipe Ingredient Status Map
For each recipe, compute:
- `ingredientStatusByName: Record<string, 'HAVE'|'MAYBE'|'NEED'>` (or your existing labels)
- Optional summary counts:
  - `haveCount`, `maybeCount`, `needCount`, `totalCount`

C) Pantry Fit Score for Sorting For You
For each recipe in For You results, compute a score used ONLY for sorting/prioritization:
- Base formula (MVP-simple, deterministic):
  - HAVE = 2 points
  - MAYBE = 1 point
  - NEED = 0 points
  - Score = sum(points) / (2 * totalCount)  → yields 0..1
- Tie-breakers (in this order):
  1) Higher haveCount
  2) Lower needCount
  3) Stable fallback: original API order (do not reshuffle unpredictably)

Then:
- Sort For You recipes by score desc (and tie-breakers), AFTER applying user preference hard filters (already in place).
- Do not change the FatSecret request for this prompt; this is client-side ranking only.

2) The same recipe displayed in multiple places must not drift:
   - If a recipe appears in For You list and is opened in detail, the statuses must match.
3) When the pantry changes (user edits pantry), the For You feed must update statuses:
   - Implement a recompute step triggered by pantry store changes.
   - Do not force a new FatSecret fetch; recompute on existing loaded recipes first.
