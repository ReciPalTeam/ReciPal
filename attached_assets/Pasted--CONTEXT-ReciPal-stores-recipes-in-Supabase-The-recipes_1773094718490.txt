
CONTEXT
ReciPal stores recipes in Supabase. The recipes table has: steps (JSONB array), servings (integer), cook_time_minutes, total_time_minutes, and cook_time_scale_type (values: invariant, linear_batch, weight_based, surface_area, or null). A recipe_steps_variants table exists with a unique constraint on (recipe_id, servings) — this is the cache. Nutrition lives in recipe_nutrition_totals with calories_per_serving, protein_per_serving, carbs_per_serving, fat_per_serving, and servings. Users can select a custom serving count in the planner. When they do, scaled steps and macros must be returned.
Each step in the steps array has this shape:
  { step: number, time: string, equipment: string, instruction: string }
---
TASK 1 — Create server/scaledSteps.ts
This function accepts recipe_id (string) and desired_servings (integer) and returns scaled steps, cook time, and macros.
Logic:
1. Query recipe_steps_variants for a row matching (recipe_id, desired_servings). If found, return it immediately with no GPT call.
2. If not cached, fetch from Supabase:
   - From recipes: steps, servings, cook_time_minutes, cook_time_scale_type
   - From recipe_nutrition_totals: calories_per_serving, protein_per_serving, carbs_per_serving, fat_per_serving, servings
3. Compute scaled nutrition using:
   scaled_macro = (macro_per_serving x stored_servings) / desired_servings
   Apply to all four macros. Round to one decimal place.
4. Compute scaled cook time based on cook_time_scale_type:
   - invariant or null: return original cook_time_minutes unchanged
   - linear_batch: Math.ceil(desired_servings / source_servings) x cook_time_minutes
   - weight_based: Math.round(cook_time_minutes x (desired_servings / source_servings))
   - surface_area: Math.round(cook_time_minutes x Math.pow(desired_servings / source_servings, 0.33))
5. Call GPT-4o with the following:
   System prompt:
   "You are a professional recipe editor. You will receive a recipe's original cooking steps, the original serving count, a new desired serving count, and a scale type. Rewrite the steps array so that all ingredient quantities mentioned in the instructions are scaled proportionally to the new serving count. Adjust cook times based on the scale_type: invariant means do not change any cook times; linear_batch means increase cook times proportionally and note batching where needed; weight_based means adjust cook times proportionally to the weight change; surface_area means make moderate time adjustments using cube root scaling and mention using a larger or smaller pan where relevant. Keep the same number of steps and the same JSON structure. Each step must have: step (integer), time (string e.g. 10 min), equipment (string), instruction (string). Return only the JSON array of steps with no explanation or markdown."
   User message:
   "Original servings: {source_servings}
   Desired servings: {desired_servings}
   Scale type: {cook_time_scale_type or invariant if null}
   Steps: {JSON.stringify(original_steps)}"
6. Parse the GPT response as a JSON array. Insert into recipe_steps_variants:
   recipe_id, servings = desired_servings, steps = parsed array, cook_time_minutes = scaled cook time.
   Use an upsert with ON CONFLICT (recipe_id, servings) DO NOTHING to avoid race conditions.
7. Return:
   { steps, cook_time_minutes, calories_per_serving, protein_per_serving, carbs_per_serving, fat_per_serving }
---
TASK 2 — Wire getScaledSteps into the planner
In client/src/pages/planner/index.tsx, when a user changes the serving count for a meal:
- Call getScaledSteps(recipe_id, desired_servings)
- Replace the displayed steps with the returned steps
- Replace the displayed cook time with the returned cook_time_minutes
- Replace the displayed macros with the scaled nutrition values
- Show a loading state on the recipe card while the call is in progress (cache misses take 2-4 seconds)
- On error, keep the original steps and macros displayed and log the error to the console
---
TASK 3 — Expose an API endpoint to trigger cook_time_scale_type classification
Create a server endpoint (e.g. POST /api/classify-cook-time-scale) that runs the following classification job. This endpoint is called manually — it does not run on app start.
The job:
1. Fetch all recipes where cook_time_scale_type IS NULL, in batches of 50
2. For each recipe, send the title and steps to GPT-4o with this prompt:
   System prompt:
   "Classify this recipe's cook time scaling behavior into exactly one of these four categories: invariant (total cook time does not meaningfully change with serving count — soups, stews, braises, curries, sauces), linear_batch (more servings requires more batches at the same time per batch — pan-fried items, stir-fry, tacos, cookies, pancakes), weight_based (cook time scales with total ingredient weight — roasts, whole birds, whole fish), surface_area (cook time increases moderately with larger pan or dish size — lasagna, casseroles, sheet cakes, quiche, gratin). Respond with exactly one word: invariant, linear_batch, weight_based, or surface_area."
   User message:
   "Recipe title: {title}
   Steps: {JSON.stringify(steps)}"
3. Update recipes.cook_time_scale_type with the GPT result for each recipe
4. Return a JSON response with counts: { classified: number, skipped: number, errors: number }
---
ACCEPTANCE CRITERIA
- Changing serving count in the planner calls getScaledSteps and updates steps, cook time, and macros
- A second request for the same (recipe_id, servings) returns instantly from cache with no GPT call
- Macros scale correctly using the formula above
- Cook time updates according to the correct formula for each scale type
- Loading state is visible during GPT generation and clears on completion
- Errors do not crash the planner UI
- POST /api/classify-cook-time-scale runs successfully and updates cook_time_scale_type for all unclassified recipes
