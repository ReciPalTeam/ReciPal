CONTEXT
- Feature: Planner → “Preview Your Week” window/modal → “Auto-populate Week” card/functionality.
- Current problem: Recipe fetching is wrong and appears to use old/incorrect logic (including “most recent 200 recipes”).
- We must change fetching to a new protocol:
  A) Fetches should request 7 recipes for each meal_type on initial load.
  B) On “Regenerate” (or equivalent), fetch 7 MORE recipes per meal_type (next batch).
  C) If user increases amount shown by a set amount of servings, fetch logic must follow the same batching protocol (incremental 7-per-meal_type batches) and not re-pull the same recipes.
  D) Fetching must FIRST apply account-type preference logic:
     - Use Free user preferences OR Pro user preferences depending on subscription/account state.
     - For Pro users, enforce calorie + macro goals as HARD LIMITS (recipes returned must satisfy constraints).
  E) Do NOT fetch “most recent 200 recipes” or any large unfiltered pool.
- Data source for recipes must be Supabase (no placeholders).

NEW FETCH PROTOCOL (must implement exactly)
For a given user and week preview session:
1) Determine account tier: Free or Pro.
2) Load applicable preferences:
   - Free: use Free preference profile/fields
   - Pro: use Pro preference profile/fields AND enforce macro/calorie goals as hard constraints
3) For each meal_type in the week plan:
   - Fetch batch_size_per_meal_type (=7) recipes from Supabase that match the applicable preferences.
   - Pro users: recipes MUST satisfy calorie + macro constraints (hard filter).
4) Maintain a “seen recipe ids” set per meal_type per session to prevent repeats.
5) On regenerate:
   - For each meal_type, fetch the NEXT 7 recipes that match constraints and are not in seen set.
   - Append/replace UI per existing behavior (match current UI behavior; do not change UX semantics).
6) On servings increment behavior:
   - If the UI requires showing more recipe options or generating more planned meals due to serving changes, request additional recipes using the same “next 7 per meal_type” approach, respecting the seen set and constraints.

SUPABASE QUERY REQUIREMENTS
- Query must be constraint-driven:
  - Apply preference filters first (cuisines, exclusions, diet tags, equipment, etc. based on existing schema).
  - Pro: apply macro/calorie hard constraints in the query when possible; otherwise apply in code immediately after fetch with strict rejection and then continue fetching until batch is filled or a safe cap is hit.
- Pagination strategy:
  - Use stable ordering that supports pagination without “most recent 200” (e.g., deterministic ordering by a rank score or created_at with cursor) BUT still must not pull huge pools.
  - Use limit sized to fill the batch efficiently. If post-filtering in code is required, fetch small pages iteratively until you fill the batch (or hit cap).
- De-duplication:
  - Must exclude seen recipe ids (server-side if possible; otherwise client-side plus iterative fetching).
- Safety cap:
  - Implement a hard cap on query attempts to avoid infinite loops when constraints are too tight (e.g., 10 iterations). If unable to fill, show partial results with a clear empty-state note (“Not enough matching recipes; adjust preferences”).
