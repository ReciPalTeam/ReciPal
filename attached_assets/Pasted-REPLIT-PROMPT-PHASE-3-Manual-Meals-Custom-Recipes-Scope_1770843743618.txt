PHASE 3 — MANUAL MEALS & CUSTOM RECIPES (BIG LIFT)
========================================================

7) Manual Meal = Ingredient-Based (Not Flat Calories)
Goal: Manual entry must evolve from “Enter calories/macros” to “Build meal from ingredients”.

Required behavior
- Keep existing fields: calendar selection (day / date range), meal slot selection, and meal name.
- Add new sections to manual entry UI to support ingredient-based construction:
  A) Ingredient list builder (add ingredients one-by-one)
  B) For each ingredient entry, capture at minimum:
     - ingredient name/query (string)
     - amount/quantity (number)
     - unit (string; use existing unit system if present; otherwise a minimal unit picker)
  C) Each ingredient added must:
     - Call FatSecret API
     - Pull full nutrition data for that ingredient (calories + macros at minimum; and any other nutrients available from the API response should be stored if already supported)
- System must:
  - Sum all ingredient nutrients into a single meal total
  - Create a custom meal record from the summed nutrients
  - Save it as a reusable custom recipe for that user

Implementation notes (do not over-engineer)
- If the current FatSecret integration already supports ingredient search + nutrition details, reuse it.
- If FatSecret returns multiple matches, select the best match deterministically (e.g., first/highest confidence) BUT also allow the user to change the selected match if the current UI patterns already support selection. If not, implement the smallest “choose from top N results” modal/list.
- Cache per-ingredient lookup results for the current session to avoid repeated API calls while editing (in-memory only unless there is an existing caching layer).
- Do not require barcode/receipt/photo flows for this phase. Only ingredient-based manual meals.

8) Custom Recipe Creation & Storage
When a manual meal is saved:
- Create a recipe card for the custom recipe
- Use a placeholder image (ReciPal logo) — no user photo required
- Auto-add to:
  A) Planner (on the correct day(s) + correct meal slot)
  B) User’s recipe library (so it can be reused like a normal recipe)
- Ensure that “save” is idempotent enough to avoid duplicates if user double-clicks (disable button while saving and/or server-side guard if available).

9) “My Meals” / “My Recipes” Section
- Rename the existing “Favorites” tab to “My Meals” (rename only where displayed; do not break underlying keys unless necessary).
- Inside “My Meals”, add a toggle:
  - Favorites | My Recipes
- Custom recipes created from manual meals:
  - Live under “My Recipes”
  - Are reusable like any normal recipe (can add to planner, etc.)
  - Are editable later
  - No images required beyond the placeholder (ReciPal logo is fine)

Editing requirements (minimum viable)
- Provide an “Edit” action for custom recipes that allows:
  - Editing the name
  - Editing the ingredient list (add/remove/update qty/unit)
  - Recalculate totals by re-calling FatSecret for changed/new ingredients (reuse cached results where possible)
- If the codebase has a standard edit modal/page pattern, follow it. If not, implement the simplest modal.

10) Pantry Integration for Manual Meals
When a custom meal is logged (i.e., added to planner AND marked/treated as consumed in the same way normal meals affect pantry today—follow the existing “consumption” trigger behavior):
For each ingredient in the custom meal:
- If the ingredient exists in pantry → reduce quantity based on usage
- If it does NOT exist in pantry → do nothing (no prompt, no warnings)
- Pantry must always reflect actual consumption for manual meals

Matching rules (keep it simple + consistent)
- Use the existing pantry item matching strategy if one exists.
- If none exists: match case-insensitively on a normalized name (trim, lower, collapse whitespace). Do NOT introduce fuzzy matching unless the project already uses it.
