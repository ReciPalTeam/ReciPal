PHASE 1 (SERVER API): Replace FatSecret recipe FEED/DATA source with Supabase via server-only endpoints (no client Supabase access)

NON-NEGOTIABLE CONSTRAINTS
- Do NOT add any Supabase keys (anon or service role) to the client. The browser must never talk to Supabase directly.
- All Supabase access must be server-side only using SUPABASE_URL and SUPABASE_SERVICE_ROLE_KEY from Replit Secrets.
- Manual Entry ingredient search (FatSecret) must remain untouched.
- Do not change For You personalization/filtering logic; only change the data source so the same logic runs against Supabase-provided recipes.
- Free/Pro users (and everyone except admin) must never be able to access secrets, API keys, or raw DB credentials. Do not log secrets.

GOAL
Create server endpoints that return the full recipe payload from Supabase:
- recipes
- recipe_nutrition_totals
- recipe_ingredients
- ingredients
- ingredient_nutrients
So the frontend can render feeds and recipe cards WITHOUT calling FatSecret for recipes.

REQUIRED ENDPOINTS
1) GET /api/recipes/feed/for-you
- Requires authenticated user (same auth system currently used in the app).
- Returns a list of recipes for For You using the EXISTING filtering logic (do not rewrite the algorithm; just swap where it fetches recipe data from).
- Response must include enough fields to render the existing recipe cards (title/name, image, calories/macros summary, etc.) plus recipe_id.

2) GET /api/recipes/feed/something-new
- Requires authenticated user.
- Returns newest/global recipes unfiltered.
- Include same card-ready fields as For You.

3) GET /api/recipes/:recipeId
- Requires authenticated user.
- Returns ONE recipe with FULL nested payload:
  - recipe
  - recipe_nutrition_totals
  - recipe_ingredients (with quantities/units/display text)
  - ingredients
  - ingredient_nutrients
- Shape it so the existing “expanded recipe card / recipe detail” UI can render without additional client-side joins.

IMPLEMENTATION DETAILS
- Add a server-side Supabase client helper (e.g., server/lib/supabaseServer.ts) that uses SUPABASE_SERVICE_ROLE_KEY.
- Add a server-side “data access layer” for recipes (e.g., server/lib/recipeDb.ts) with functions:
  - getForYouFeed(userId, userPrefs)
  - getSomethingNewFeed()
  - getRecipeById(recipeId)
- Minimize round trips: prefer a small number of queries that fetch all required related rows. If needed, do multiple queries but keep it clean and deterministic.
- Add input validation:
  - recipeId must be a valid id type used by the DB
- Add safe error handling:
  - 401 if not authenticated
  - 404 if recipe not found
  - 500 with a generic message if Supabase fails (no secret leakage)
- Logging:
  - Log endpoint name + correlationId + status only. Never log keys or full DB responses.

FATSECRET FAILSAFE (SERVER ONLY)
- Add an optional fallback inside getRecipeById:
  - If Supabase returns “not found” OR Supabase is down, and the app previously supported loading that recipe from FatSecret, then attempt FatSecret as fallback.
  - This fallback must be server-side and must not change the primary feed sources (feeds remain Supabase).
  - If fallback succeeds, return a response shaped EXACTLY like the Supabase payload shape expected by the UI (so the UI doesn’t care where it came from).

ACCEPTANCE CRITERIA (MUST ALL PASS)
- The browser network tab shows NO calls to Supabase domains and contains NO Supabase keys.
- For You feed and Something New feed load using the new endpoints (server -> Supabase).
- Clicking a recipe opens the expanded view using GET /api/recipes/:recipeId and renders correctly.
- Manual Entry ingredient search still uses FatSecret exactly as before.
- No secrets appear in logs or responses.

VERIFICATION STEPS (DO THESE AFTER CODING)
1) Run the app, open DevTools Network:
   - Confirm feeds call /api/recipes/feed/* and recipe detail calls /api/recipes/:id
   - Confirm there are ZERO requests to Supabase from the browser
2) Click 3 recipes from For You and 3 from Something New:
   - Confirm card + expanded view render and macros/calories display
3) Temporarily break SUPABASE_URL (simulate failure) and load a known recipe that used to exist via FatSecret:
   - Confirm getRecipeById can still return via fallback (server-only), OR returns a clean 500/404 with no leak
