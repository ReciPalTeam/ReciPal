ReciPal FatSecret Integration Plan (Replit Execution Checklist)

Goal:
Replace mockRecipes as the recipe data source with FatSecret-backed recipes WITHOUT breaking existing UI. Preserve the canonical Recipe contract in client/src/lib/mock-data.ts exactly as-is. Do NOT integrate Instacart yet.

1) Configure Secrets (NO hardcoding)
- Add env vars in Replit Secrets:
  - FATSECRET_CLIENT_ID
  - FATSECRET_CLIENT_SECRET
- Code must only read from process.env.* (never paste real keys into files).

2) Build FatSecret server client (OAuth2 + safe request helper)
- Implement OAuth2 client-credentials token fetch.
- Cache token in memory with expiry, refresh ~5 minutes before expiration.
- Create a single helper fatsecretCall(params) that:
  - attaches Authorization: Bearer <token>
  - calls the FatSecret API endpoint(s) needed.

3) Build the adapter: FatSecret → canonical Recipe (mockRecipes shape)
- Create a pure mapping function that converts FatSecret recipe.get response into:
  {
    id, title, image, cookTime, servings,
    calories, protein, carbs, fat,          // PER SERVING numbers
    mealTypes, cookingStyle,
    ingredients: [{ name, amount (string), unit }],
    steps: string[]
  }
Mapping rules:
- cookTime = "<prep_time_min + cooking_time_min> min" (Option A); if missing => "—"
- Macros MUST be per-serving (planner multiplies by meal.servings).
- Ingredient.amount MUST remain a string (cart parses parseFloat(amount)||1 later).
- Steps must be ordered by step number.

4) Create API endpoints (server-side only; client never calls FatSecret directly)
- GET /api/recipes/search
  - supports limit (default 20), page (default 0), q optional
  - uses FatSecret recipes.search to get recipe_ids (must_have_images=true)
  - hydrates each recipe_id using recipe.get
  - maps each full recipe through the adapter
  - returns { recipes: Recipe[], page, limit }
- GET /api/recipes/:id
  - calls recipe.get for that recipe_id
  - maps through adapter
  - returns { recipe: Recipe }

5) Add caching to control call volume
- Token cache (from step 2).
- recipe.get cache keyed by recipe_id for ~6 hours.
- search cache keyed by (q,limit,page) for ~5 minutes.
- Ensure caching does not break when server reloads (in-memory is fine for now).

6) Wire client to use the new endpoints (minimal changes, no refactors)
- Feed page:
  - Replace mockRecipes array source with fetch from /api/recipes/search
  - Initial load: 20 recipes
  - Infinite scroll: load more in batches of 5
  - Dedupe by recipe.id and prevent double-fetch while loading
  - Keep existing render usage: recipe.image, recipe.cookTime, recipe.calories/protein/carbs/fat, etc.
- Recipe detail page:
  - Replace mockRecipes.find(...) with fetch /api/recipes/:id
  - Preserve UI display fields exactly (title, image, cookTime string, servings, macros, ingredients, steps)
- Planner page:
  - Keep join logic by recipeId, but use a shared client cache/store keyed by recipeId
  - Avoid excessive refetching

7) Ensure filters don’t break (minimum viable classification)
- Because FatSecret won’t always supply ReciPal taxonomy, adapter must deterministically derive:
  - mealTypes (Breakfast/Lunch/Dinner/Snack) via keyword rules
  - cookingStyle via keyword rules, else default "Healthy / Light"
- Keep deterministic + stable; do not introduce randomness.

8) Validation / Acceptance Checks (must pass)
- Feed loads 20 FatSecret recipes with images and renders without errors.
- Infinite scroll loads additional recipes in +5 batches without duplicates.
- Recipe detail shows steps + ingredients and macros.
- Planner macro totals still compute correctly (per-serving macros * meal.servings).
- Cart add continues to work (ingredient.amount remains string; parseFloat used downstream).
- No secrets appear in client bundles or logs.

Output requirement:
- List files created/edited.
- Provide code for created files.
- Provide minimal diffs for edited files.
- No unrelated refactors or schema changes.
