This is PROMPT P4 of 4 for the ReciPal codebase. Prompts P1, P2, and P3 are complete.

SCOPE (HARD-LOCK)
- This prompt ONLY implements ingredient-level swap UX and logic inside Recipe Detail and Planner meal detail.
- Do NOT change Recipes feed cards.
- Do NOT change Planner totals logic (P1).
- Do NOT change Auto-populate preview overlay (P2).
- Do NOT change Add-to-Plan scheduling popup (P3).
- Do NOT change Pantry, Cart, Profile, Settings, Paywall, or RevenueCat logic.
- Do NOT change Need / Might Have / Have UI labels or behavior.

GOAL
Implement ingredient-level swapping for meals with:
- Clear ingredient classification labels
- Inline Swap buttons next to each ingredient
- Dedicated swap popup with:
  - 4 suggested replacements
  - Search bar
  - Regenerate suggestions
- Dynamic nutrition recalculation
- No change to recipe instructions text
- No breaking of existing Recipe Detail or Planner layouts

This feature is foundational for long-term intelligence, personalization, and Pro logic.

────────────────────────────────────────
A) INGREDIENT CLASSIFICATION (VISIBLE LABELS)
────────────────────────────────────────

Each ingredient in a recipe must receive a visible classification label:

- Protein
- Carb
- Veggie
- Fruit
- (Fallback: Other)

These labels are NOT nutrition macros. They are ingredient-type groupings used for swap logic and UX clarity.

Rules:
- These labels appear inline next to each ingredient in the ingredient list.
- They must NOT replace or interfere with:
  - Need
  - Might Have
  - Have
- Both labels can coexist (e.g., “Have • Protein”).

Classification method (MVP heuristic):
- Protein: meat, poultry, fish, eggs, tofu, tempeh, beans, legumes, protein powders, Greek yogurt
- Carb: bread, rice, pasta, potatoes, oats, tortillas, grains, sugar
- Veggie: vegetables, leafy greens, mushrooms, etc.
- Fruit: fruit items
- Other: oils, spices, condiments, baking ingredients, etc.

This logic must be implemented in a centralized helper so it can be reused later when APIs are added.

────────────────────────────────────────
B) SWAP BUTTONS (INLINE)
────────────────────────────────────────

For every ingredient in the ingredient list:

- Add a “Swap” button inline next to that ingredient.
- Do NOT remove or relocate any existing UI elements.
- Do NOT modify the layout of the ingredient list beyond adding the Swap button and the classification label.

────────────────────────────────────────
C) SWAP POPUP UI
────────────────────────────────────────

When the user taps Swap:

Open a dedicated modal (not the Add-to-Plan popup) containing:

Header: “Swap Ingredient”
Subheader: “Choose a replacement”

Main content:
1) Suggested replacements (4 items)
2) Search bar at top: “Search ingredients”
3) Regenerate button (refreshes suggestions)

Each suggestion item shows:
- Ingredient name
- Pantry overlap indicator (if applicable)
- Classification label (Protein/Carb/Veggie/Fruit/Other)

The popup must feel lightweight and fast.

────────────────────────────────────────
D) SUGGESTION LOGIC
────────────────────────────────────────

Suggested replacements must:

Hard constraints:
- Match the same classification type whenever possible
  (Protein → Protein, Carb → Carb, etc.)
- Respect:
  - Allergies
  - Dietary restrictions
  - Disliked ingredients (hard exclusion)

Ranking boosts:
- Pantry overlap
- Favorites
- Frequently used ingredients
- Cost preference match
- Cooking comfort match

Pro-only boosts:
- Macro alignment
- Calorie alignment
- Goal gap closure (if protein low, suggest higher protein replacements, etc.)

If no good same-category replacement exists:
- Fall back to “Other” category
- Still respect allergies/diet/dislikes

────────────────────────────────────────
E) SWAP BEHAVIOR
────────────────────────────────────────

When the user selects a replacement:

- Replace ONLY the ingredient line
- Do NOT modify the recipe instructions text
- Do NOT attempt to rewrite cooking steps
- Do NOT change recipe name

Update:
- Ingredient list
- Nutrition totals (calories, protein, carbs, fat)
- Pantry overlap indicators
- Need / Might Have / Have logic

Show a subtle toast:
“Ingredient updated”

────────────────────────────────────────
F) NUTRITION RECALCULATION
────────────────────────────────────────

After each swap:

- Recalculate nutrition dynamically
- Update:
  - Calories
  - Protein
  - Carbs
  - Fat

Rules:
- If exact nutrition is unknown, use best-effort heuristics.
- If macros are unknown, recalc calories only.
- Do NOT block the UI due to missing nutrition data.

For Pro users:
- This recalc should affect projected totals in Planner preview (P2)
- This recalc should affect macro alignment logic

For Free users:
- Still recalc calories and show updated values
- No goal language

────────────────────────────────────────
G) DATA MODEL REQUIREMENTS
────────────────────────────────────────

Swapped ingredients must persist:

Each planned meal entry must store:
- Original recipeId
- Ingredient overrides:
  - originalIngredientId
  - replacementIngredientId
  - replacementName
  - replacementNutrition (if known)

This ensures:
- Swaps persist across sessions
- Different days can diverge
- Entries are not linked (per P3)

────────────────────────────────────────
H) PLANNER INTEGRATION
────────────────────────────────────────

When viewing a planned meal inside the Planner:

- The ingredient list should reflect swaps.
- Swap buttons should still be available.
- Swapping inside Planner modifies only that entry, not other identical ones.

Swaps must:
- Update that meal’s nutrition
- Affect totals when counted (per P1 logic)
- Not propagate to other days

────────────────────────────────────────
I) PERFORMANCE + UX CONSTRAINTS
────────────────────────────────────────

- Swap popup must open in under 200ms.
- No page reloads.
- No full recipe rerenders.
- No navigation away from the current screen.
