CONTEXT
ReciPal recipe detail pages and MealDetailPopup have a serving counter (minus/plus buttons). Each recipe has a base serving count in recipes.servings (e.g. Calentado = 4). Three bugs exist and all three must be fixed in this task.

TASK 1 — Fix nutrition display: always show TOTAL calories and macros, never per-serving
Files: client/src/pages/recipe/[id].tsx, client/src/components/meal-detail-popup.tsx
What's happening now: The page loads and briefly flashes the per-serving value (617 cal), then re-renders to a different value. The display is wrong in both states.
What should happen: On first render the page immediately displays the TOTAL nutrition for the base serving count. It never displays per-serving values anywhere. There is no flash, no re-render, no useEffect that corrects an initially wrong value.
How to fix:
1. Initialize the serving counter state to recipe.servings (the base count from Supabase). This is the floor — the counter can never go below this value.
2. Initialize displayed nutrition as the INITIAL state value (not in a useEffect):
   displayed_calories = Math.round(nutrition.calories_per_serving * recipe.servings)
   displayed_protein  = Math.round(nutrition.protein_per_serving  * recipe.servings)
   displayed_carbs    = Math.round(nutrition.carbs_per_serving    * recipe.servings)
   displayed_fat      = Math.round(nutrition.fat_per_serving      * recipe.servings)
3. When the serving counter changes to any value N:
   displayed_calories = Math.round(nutrition.calories_per_serving * N)
   displayed_protein  = Math.round(nutrition.protein_per_serving  * N)
   displayed_carbs    = Math.round(nutrition.carbs_per_serving    * N)
   displayed_fat      = Math.round(nutrition.fat_per_serving      * N)
4. The formula is ALWAYS: per_serving_value * current_serving_count. This is multiplication only. There is no division anywhere in the nutrition math. Calories and macros always increase when servings increase and always decrease when servings decrease. They can never go below the base total (per_serving * base_servings).
5. Display label format: "X cal · Xg P · Xg C · Xg F for [N] servings"
6. Remove any existing logic that divides, inverts, or shows per-serving nutrition.
Acceptance: Calentado loads showing "2469 cal · 102g P · 270g C · 107g F for 4 servings" immediately with zero flash.

TASK 2 — Fix serving counter: count by 1, floor at base servings, style the disabled minus button
Files: client/src/pages/recipe/[id].tsx, client/src/components/meal-detail-popup.tsx
What's happening now: The counter jumps from 4 to 8 (incrementing by the base serving count instead of by 1). This is wrong.
What should happen: The counter always increments and decrements by exactly 1 serving. The floor is the recipe's base serving count. For Calentado (base 4): 4 → 5 → 6 → 7 → 8. For a single-serve recipe (base 1): 1 → 2 → 3 → 4.

2. Minus button: servings = Math.max(servings - 1, recipe.servings)
3. The counter can never go below recipe.servings (the base floor).
4. Minus button is disabled when servings === recipe.servings.
5. When the minus button is disabled, apply these Tailwind classes: opacity-30 cursor-not-allowed. Do not rely only on the HTML disabled attribute — make the disabled state visually obvious.
6. When the minus button is enabled, remove opacity-30 and cursor-not-allowed so it looks fully clickable.
Acceptance: Calentado counter goes 4 → 5 → 6 → 7 → 8 on plus clicks. Minus at 4 is greyed out and unclickable.

TASK 3 — Fix ingredient scaling: GPT must return scaled ingredients, and the UI must display them
Files: server/scaledSteps.ts, client/src/pages/recipe/[id].tsx, client/src/components/meal-detail-popup.tsx
What's happening now: When servings change, the ingredient list in the Ingredients tab does not update. Ingredients still show the original base amounts regardless of the serving count.
What should happen: Every time the serving counter changes away from the base value, a GPT call returns scaled ingredients AND steps. The Ingredients tab immediately displays the scaled ingredient amounts. When servings return to the base value, the original ingredients from Supabase are shown.
How to fix:
3a. Update the getScaledSteps function to return two keys in the GPT response: "steps" and "ingredients".
3b. Update the GPT system prompt in server/scaledSteps.ts to this exact text:
"You are a professional recipe editor. You receive a recipe's original cooking steps, original ingredient list, the original serving count, a new desired serving count, and a scale type. Return a JSON object with two keys: steps and ingredients.
For steps: rewrite each step so all quantities are scaled proportionally to the new serving count. Adjust cook times based on scale_type — invariant means keep cook times the same; linear_batch means increase cook times proportionally and note batching; weight_based means adjust cook times proportionally to weight change; surface_area means make moderate time adjustments using cube root scaling and mention pan size changes. Each step object has: step (integer), time (string like 10 min), equipment (string), instruction (string).
For ingredients: return each ingredient with its amount scaled proportionally to the new serving count. Express all amounts as common fractions or practical measurements — never raw decimals. Use formats like 1/4, 1/2, 3/4, 1 1/3, 1 1/2, 2, 3 oz. Each ingredient object has: display_text (string — the full human-readable ingredient line like 1 1/2 cups rice), amount (number — the numeric value), unit (string — the unit of measurement).
Return only the JSON object. No explanation, no markdown."
3c. Update the GPT user message to include the ingredient list:
"Original servings: {source_servings}
Desired servings: {desired_servings}
Scale type: {cook_time_scale_type or invariant if null}
Ingredients: {JSON.stringify(original_ingredients)}
Steps: {JSON.stringify(original_steps)}"
Where original_ingredients is fetched from recipe_ingredients for the recipe_id, returning display_text, amount, and unit for each row ordered by sort_order.
3d. Create a new table recipe_ingredients_variants to cache scaled ingredients (mirrors recipe_steps_variants pattern for steps). Run this SQL:
This table already exists in Supabase — do not recreate it. Its schema:
  recipe_ingredients_variants (
    variant_id UUID PRIMARY KEY,
    recipe_id UUID NOT NULL REFERENCES recipes(recipe_id) ON DELETE CASCADE,
    servings INTEGER NOT NULL,
    ingredients JSONB NOT NULL,
    created_at TIMESTAMPTZ,
    updated_at TIMESTAMPTZ,
    UNIQUE(recipe_id, servings)
  )
  Indexed on (recipe_id, servings).
The ingredients JSONB column stores an array of objects, each with: sort_order (integer matching recipe_ingredients.sort_order), display_text (string), amount (number), unit (string).
3e. Cache read/write flow for ingredient variants:
On serving count change (away from base):
  1. Check recipe_ingredients_variants for a row matching (recipe_id, servings).
  2. If cache hit: use the cached ingredients JSONB. Do not call GPT.
  3. If cache miss: call GPT (which returns both steps and ingredients). Write the scaled steps to recipe_steps_variants AND write the scaled ingredients to recipe_ingredients_variants. Use UPSERT (ON CONFLICT (recipe_id, servings) DO UPDATE) for both writes.
On serving count return to base:
  Do not read from cache. Show original ingredients from recipe_ingredients table.
3f. On the recipe detail page and MealDetailPopup — rendering scaled ingredients:
  - When scaled ingredients are returned (from cache or GPT), merge them onto the original Supabase ingredient objects by matching sort_order. Each original ingredient keeps its id, name, have_status, sort_order, and Swap button. Only the amount and unit values are replaced with scaled values.
  - Render using the SAME ingredient card component used for base ingredients. Scaled ingredients must show Have/Need badges, Swap buttons, and full card styling. There is never a separate plain-text render path for scaled ingredients.
  - When servings return to base, pass the original Supabase ingredients with no modifications.
  - Show a loading spinner on the Ingredients tab while waiting for GPT response on cache miss.
Acceptance: Calentado at 5 servings shows 25% more of each ingredient (e.g. if base has 3 cups rice at 4 servings, 5 servings shows 3 3/4 cups rice). At 6, 7, 8 servings amounts continue scaling proportionally. Returning to 4 servings restores original amounts. Ingredient cards always retain Have/Need badges, Swap buttons, and full styling at every serving count. No raw decimals and no unstyled plain text ever appear.
