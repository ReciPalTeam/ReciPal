SCOPE (HARD-LOCK)
- Only modify the “Add to Plan” experience and scheduling logic.
- Do NOT change the Recipes feed card UI layout or styling.
- Do NOT change Planner totals logic (P1) or Auto-populate preview overlay (P2) except where needed to integrate scheduling output.

GOAL
Redesign the Add-to-Plan flow so scheduling is centralized, explicit, and powerful:
- “Add to Plan” from recipe cards continues to work without changing recipe card UI.
- When the user enters Recipe Detail (or meal card detail view), clicking “Add to Plan” inside that detail opens a scheduling popup (not a multi-step wizard).
- Scheduling popup supports:
  1) select a single day
  2) select a continuous date range
  3) select multiple specific days (Mon/Wed/Fri)
- User must choose a meal slot for placement:
  Breakfast, Lunch, Dinner, Desserts, Snackitizers
- Popup calendar must visually show which days/slots are already filled.
- If selected day/slot is already filled, allow replacing but show a warning confirmation first.
- On confirm:
  - distribute IDENTICAL entries across selected days (not linked afterwards)
  - write into the real Planner calendar immediately
  - entries start in “scheduled” state (per P1)
  - do NOT auto-mark cooked
- No UI breaks, no card layout changes.

C) REPLACEMENT WARNING (REQUIRED)
────────────────────────────────────────

If the user selects day(s) and the chosen slot is already filled for any of those days:

- Before saving, show a confirmation dialog:
  Title: “Replace existing meal?”
  Body: “One or more selected days already has a meal in this slot. Replacing will remove the existing meal from the plan.”
  Buttons:
   - Primary: “Replace”
   - Secondary: “Go Back”

Behavior:
- If Replace: overwrite only those conflicting slots the user selected.
- If Go Back: return to popup without changes.

Note:
- Replacement must only apply to days the user selected.
- Do not affect other days.

D) WRITE-TO-PLANNER RULES (CRITICAL)
────────────────────────────────────────

When user confirms Add to Plan:

D1) Create planner entries for each selected day in the chosen slot.
- Each entry must be identical in content at the time of creation (same recipeId, same servings, same any current recipe state).
- Entries are NOT linked after creation. Editing one later does not alter the others.

D2) Store planned entry fields at minimum:
- id
- userId
- date (YYYY-MM-DD)
- slotType (breakfast/lunch/dinner/dessert/snackitizer)
- recipeId
- servings

- status: "scheduled" initially

D3) Do NOT affect consumption totals immediately.
- Consumption totals are counted only when cooked or auto-counted at midnight (P1 rules).

D4) The planner calendar UI must update immediately to reflect newly scheduled entries.

D5) Ensure this does not interfere with the Auto-populate preview overlay (P2).
- Auto-populate still writes only to empty slots on confirm.
- Manual Add-to-Plan writes to selected slots and can replace with warning.
