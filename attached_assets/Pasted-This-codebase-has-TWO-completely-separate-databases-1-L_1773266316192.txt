This codebase has TWO completely separate databases:

1. LOCAL PostgreSQL managed by Drizzle ORM — defined in shared/schema.ts with serial 
   integer PKs. Used for all user data: authentication, profiles, meal plans, 
   consumption logs, pantry, favorites, custom recipes, stores, savings.
   DO NOT remove, rename, or restructure any of these existing local tables.

2. Supabase recipe catalog — accessed exclusively via server/lib/recipeDb.ts using 
   the Supabase JS client. These are NOT Drizzle-managed tables and must NOT be 
   added as pgTable definitions to shared/schema.ts.

Make exactly two surgical changes to shared/schema.ts:

CHANGE 1 — Remove the isPro field from the users table:
  Delete the line: isPro: boolean("is_pro").default(false).notNull()
  The users table already derives Pro status via userProfiles.subscriptionTier — 
  isPro is redundant and causes the two-field conflict documented in the codebase.
  After removing it from the table definition, find every reference to users.isPro 
  or user.isPro anywhere in server/routes.ts, server/storage.ts, and any client files, 
  and replace each one with a lookup or reference to userProfiles.subscriptionTier === 'pro'.
  Update the InsertUser type and insertUserSchema accordingly.

CHANGE 2 — Rename maxCarbPercent to maxCarbGrams in userProfiles:
  In the userProfiles table definition, rename:
    maxCarbPercent: integer("max_carb_percent")
  to:
    maxCarbGrams: integer("max_carb_grams")
  The inline comment already documents that this column stores grams (0–999), 
  not a percentage. The column name is misleading and must be corrected.
  After renaming it in the table definition, find every reference to maxCarbPercent 
  or max_carb_percent across all files (server/routes.ts, server/storage.ts, 
  client files, shared files) and update them to maxCarbGrams / max_carb_grams.
  Update the UserProfile type and insertUserProfileSchema accordingly.
  NOTE: This renames the column in the Drizzle schema only. A matching database 
  migration (ALTER TABLE user_profiles RENAME COLUMN max_carb_percent TO max_carb_grams) 
  will be run separately — do not generate a migration file.

Then, create a new file shared/supabase-types.ts containing ONLY TypeScript interface 
definitions (no Drizzle imports, no pgTable, no createInsertSchema) that describe 
the shape of rows returned from the Supabase recipe database. These are read-only 
type definitions used to replace the `any` types throughout server/lib/recipeDb.ts.
