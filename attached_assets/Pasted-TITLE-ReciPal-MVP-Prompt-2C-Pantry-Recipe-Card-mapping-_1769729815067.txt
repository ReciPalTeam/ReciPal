TITLE: ReciPal MVP Prompt 2C — Pantry ↔ Recipe Card mapping (Have / Maybe / Need) + “For You” pantry-aware filtering

GUARDRAILS (non-negotiable):
- Do NOT touch infinite scroll batching logic from Prompt 2A/2B except to ensure pantry status is computed for newly appended recipes too.
- Do NOT change FatSecret client endpoints.
- Do NOT refactor file structure, rename modules, or add libraries.
- Do NOT implement OpenAI re-ranking (leave hooks only if explicitly described below).
- Keep changes MVP-only: pantry mapping + correct status display + For You pantry-aware result ordering/filtering.

GOAL (what must be true after this prompt):
1) Every recipe card in BOTH feeds must accurately display ingredient status:
   - HAVE / MAYBE / NEED
   based on what’s currently in the user’s Pantry.
2) “For You” feed MUST always prioritize user preferences AND pantry availability:
   - Pantry availability is an additional constraint/priority on top of existing user preference filters.
   - “Something New” is NOT pantry-bound for search results (it can still display HAVE/MAYBE/NEED labels on cards, but it should NOT use pantry to filter/priority-rank results).
3) This logic MUST apply to:
   - the initial 20 cards
   - every subsequent appended batch of 20 (infinite scroll)
4) Status display must update immediately when Pantry changes (add/remove/update ingredient state) without requiring a full page refresh.

DEFINITIONS (lock these in exactly):
- Pantry item states: HAVE, MAYBE, NEED
- A recipe ingredient is:
  - HAVE if pantry has that ingredient marked HAVE
  - MAYBE if pantry has that ingredient marked MAYBE
  - NEED if pantry does not contain that ingredient OR it is explicitly marked NEED
- Each recipe card must show a compact summary:
  - Have: X
  - Maybe: Y
  - Need: Z
  (counts of ingredients by status)

IMPORTANT CONSTRAINT:
FatSecret ingredients are strings. Pantry items may be normalized differently.
You MUST implement a lightweight normalization + matching method (below) to reduce false mismatches.

MATCHING / NORMALIZATION RULES (MVP, deterministic):
Implement a shared helper that converts ingredient names into a normalized token string:
- lowercase
- trim
- remove punctuation
- remove extra spaces
- remove common quantity/unit words if present at start (e.g., “1”, “1/2”, “tbsp”, “tsp”, “cup”, “cups”, “oz”, “lb”, “g”, “kg”, “ml”, “l”, “pinch”, “dash”, “slice”, “slices”)
- remove common preparation adjectives: “fresh”, “chopped”, “diced”, “minced”, “ground”, “boneless”, “skinless”, “frozen”
- singularize basic plurals by removing a trailing “s” when length > 3 (simple rule; no heavy NLP)
Then match using:
- exact normalized string match OR
- pantryNormalized is contained in ingredientNormalized OR ingredientNormalized is contained in pantryNormalized
Do NOT introduce fuzzy libraries. This must be pure TS.

FILES TO INSPECT / LIKELY TOUCH:
- client/src/lib/pantry-store.ts (or wherever Pantry state lives)
- client/src/lib/recipe-store.ts (feed data + recipe card models)
- client/src/components/RecipeCard.tsx (or equivalent) where card UI renders
- any selector/hooks used by feeds to read pantry + recipes

STEP 1 — Add shared normalization + match helpers
Create a new helper file (ONLY if there isn’t already a good place):
- client/src/lib/ingredient-normalize.ts
Export:
- normalizeIngredientName(name: string): string
- ingredientMatches(pantryNorm: string, ingredientNorm: string): boolean

Also export a function:
- buildPantryIndex(pantryItems) -> Map<string, PantryState>
  where key is normalized name and value is HAVE/MAYBE/NEED
This index must be memoized/cached in-memory and recomputed only when pantry changes.

STEP 2 — Compute per-recipe ingredient status summary (and store it)
In recipe store (or a small helper), implement:
- computeRecipePantryStatus(recipeIngredients: string[], pantryIndex: Map<string, PantryState>)
Returns:
{
  haveCount: number,
  maybeCount: number,
  needCount: number,
  perIngredientStatus: Array<{ raw: string, status: 'HAVE'|'MAYBE'|'NEED' }>
}
MVP UI needs only counts, but store perIngredientStatus too if it’s easy (helps later in recipe details).

Where to store:
- Extend the client-side Recipe model with:
  pantryStatus?: { haveCount, maybeCount, needCount }
Do NOT persist to DB in this prompt.

STEP 3 — Ensure pantry status is computed for ALL feed recipes
Whenever recipes are set/updated/appended in the store:
- Recompute pantryStatus for those recipes using current pantryIndex.
- For infinite scroll append:
  - compute pantryStatus for the newly fetched recipes before appending OR immediately after append.
- When pantry changes:
  - recompute pantryStatus for ALL currently loaded recipes in memory for both feeds.
  - This must not trigger new API calls—only local recompute.

STEP 4 — Update RecipeCard UI to display Have/Maybe/Need counts
On each card:
- Display a small line: “Have X • Maybe Y • Need Z”
- Must not break layout.
- If pantryStatus missing (shouldn’t happen after this prompt), fall back to “Have 0 • Maybe 0 • Need (ingredientCount)”.

STEP 5 — “For You” pantry-aware prioritization (MVP approach)
Do NOT over-engineer scoring. Use a simple, explicit ranking and (optional) filtering:

Compute a pantry “fit score” per recipe:
- score = (haveCount * 2) + (maybeCount * 1) - (needCount * 1)

Apply this ONLY to “For You”:
- After receiving the 20 recipes from the API call (and after dedupe), compute pantryStatus + score.
- Sort recipes descending by score.
- Keep the top 20 for display.

For infinite scroll on “For You”:
- Each appended batch should also be pantry-ranked within that batch BEFORE append.
- Then append the ranked batch to the existing list (do not re-sort the entire existing list to avoid jumping UI).

“Something New”:
- Do NOT sort by pantry score; keep whatever ordering it already has.
- Still compute and show pantryStatus counts.

STEP 6 — Confirm user preference priority remains intact
You MUST NOT weaken the existing For You “user preference” baseParams used for the API call.
This prompt only adds client-side pantry ranking AFTER the preference-based API results are retrieved.

OUTPUT / DELIVERABLE:
- Provide code changes only.
- List files changed + a short note per file.

ACCEPTANCE TESTS (must pass):
1) Open For You → 20 cards show and each card shows Have/Maybe/Need counts.
2) Modify pantry (toggle an item HAVE↔MAYBE or add/remove an ingredient) → counts update across visible cards immediately (no refresh).
3) Scroll For You to load more → 20 new cards append and also show accurate counts.
4) For You ordering: recipes with fewer NEED and more HAVE appear earlier (within each fetched batch).
