SCOPE (HARD-LOCK)
- Only modify the Planner tab’s Auto-Populate experience and its related preview overlay state.
- Do NOT change bottom navigation.
- Do NOT change Recipes feed card UI/layout.
- Do NOT change Pantry, Cart, Profile, Settings, Paywall, or RevenueCat logic.
- Do NOT change totals logic from P1 (only consume it).

- Do NOT implement the Add-to-Plan scheduling popup in this prompt (that is P3).

GOAL
Implement the “Auto-Populate Week” flow for Planner as a preview overlay that must be confirmed before writing to the calendar.

Key requirements:
1) Auto-populate button lives near top, under the Planner Summary Bar.
2) Clicking Auto-populate opens a Preview Overlay screen (not yet writing to calendar).
3) User can:
   - Set serving size per meal type (Breakfast/Lunch/Dinner/Dessert/Snackitizers)
   - Toggle inclusion via checkboxes: Add Desserts, Add Snackitizers
   - Regenerate preview
   - Tap into days/meals in preview to swap recipes
   - Confirm Plan -> writes preview into calendar, but ONLY fills EMPTY slots
4) Dessert and Snackitizers generation defaults:
   - At most 1 dessert per day (tied to Dinner day)
   - At most 1 snackitizer per day (any day)
5) Desserts and Snackitizers are NEVER auto-generated unless the user checked their boxes.

────────────────────────────────────────
A) PLANNER UI: AUTO-POPULATE ENTRY POINT
────────────────────────────────────────

A1) Add an “Auto-populate Week” button in Planner (Plan mode only) positioned:
- Under the Planner Summary Bar, above the calendar.

A2) For Pro users, Auto-populate is available.
For Free users, Auto-populate is still available (this is part of the core product). Do not gate it.

A3) Clicking Auto-populate opens the Preview Overlay (see B).

────────────────────────────────────────
B) PREVIEW OVERLAY: WEEK PLAN BUILDER
────────────────────────────────────────

B1) Preview Overlay must be its own screen/modal overlay that feels like a separate step, but does NOT alter the real Planner calendar until Confirm.

Overlay must contain:
- Header: “Preview Your Week”
- Subheader: “Confirm or regenerate before saving”
- Buttons:
  - “Confirm Plan” (primary)
  - “Regenerate” (secondary)
  - “Close” or Back (returns to Planner without saving)

B2) Auto-generate controls (top of overlay):
1) Checkboxes:
   - [ ] Add Desserts
   - [ ] Add Snackitizers

2) Serving size steppers (number stepper 1–10 with 10 displayed as “10+”):
   - Breakfast servings: stepper
   - Lunch servings: stepper
   - Dinner servings: stepper
   - Dessert servings: stepper
   - Snackitizers servings: stepper

Behavior:
- These steppers affect the projected nutrition totals shown in preview.
- Keep behavior consistent with serving size stepper used in Recipes filters (1–10, 10+ display).
- Default serving size = 1 for all.

B3) Preview week content
- Show the week in a compact planner-like layout (list or cards is fine).
- Each day must show planned slots in order:
  Breakfast, Lunch, Dinner, Desserts, Snackitizers
- In preview, Desserts and Snackitizers appear only if:
  - user checked the corresponding checkbox
  - otherwise show “Not included” or keep empty slot placeholder

B4) Preview totals display
- At top or inline, show projected totals for:
  - Daily calories (free)
  - Weekly calories (free)
  - For Pro users, also show projected macros (P/C/F)
IMPORTANT:
- This is “projected” totals (based on preview meals), not “consumed” totals from P1.
- Do not overwrite P1’s consumption totals.

────────────────────────────────────────
C) GENERATION LOGIC (MVP / DEMO MODE FRIENDLY)
────────────────────────────────────────

We do not have full APIs integrated yet. Generation must work using existing recipe dataset.

C1) Candidate selection rules
Generate meals by selecting recipes from the existing recipe list using heuristics:

Hard exclusions (must enforce):
- Allergies
- Dietary restrictions
- Missing tools (if recipe requires a tool user lacks)

Preference boosts (ranking):
- Pantry overlap (prioritize recipes user can mostly make)
- Cooking comfort match
- Cost preference match
- Favorites boost
- Frequently used signals (Cook Now clicks, Add to Plan events, favorited recipes)

C2) Meal type mapping
Use recipe tags/category fields to map recipes into:
- Breakfast
- Lunch
- Dinner
- Dessert
- Snackitizers

Important:
- “Snackitizers” is the merged label for snacks/appetizers/bites.
Map any snack/appetizer/bites recipes into Snackitizers.

C3) Default generation behavior
- Always try to fill Breakfast, Lunch, Dinner for each day of the week in preview.
- If “Add Desserts” checked:
  - add at most 1 dessert per day (associated under Desserts slot).
- If “Add Snackitizers” checked:
  - add at most 1 snackitizer per day (under Snackitizers slot).
- If there are insufficient recipes, reuse is allowed but avoid repetition where possible.

C4) Serving size effect
Serving size steppers should scale nutrition numbers:
- If servings=2, calories and macros double, etc.
- Apply scaling consistently for projected totals.
- If recipe nutrition is missing, scale calories only and leave macros blank or estimated.

────────────────────────────────────────
D) SWAP IN PREVIEW (RECIPE-LEVEL, NOT INGREDIENT SWAPS)
────────────────────────────────────────

Within the Preview Overlay, allow users to swap a meal recipe for that day/slot:

D1) Interaction
- User taps a preview meal entry -> opens a Swap Meal panel (simple modal).
- Swap Meal modal:
  - shows 3–6 suggested replacement recipes for that same slot type (e.g., Breakfast replacements)
  - has a search bar (“Search recipes”)
  - selecting a replacement immediately updates the preview week
  - close returns to preview overlay

D2) Suggestions for swap should use same ranking heuristics as generation, with strong boost for:
- favorites
- pantry overlap
- cooking comfort

NOTE:
- This swap is swapping the entire recipe assigned to that day/slot.
- Do NOT implement ingredient-level swap here (that is Prompt P4).

D3) After any swap:
- Update projected calories/macros immediately.

────────────────────────────────────────
E) CONFIRM PLAN WRITE RULES (CRITICAL)
────────────────────────────────────────

When user taps “Confirm Plan”:
- Write preview meals into the real Planner calendar for that week
- BUT ONLY fill EMPTY slots for selected days (the entire week is selected here)

Rules:
1) If a day/slot is already occupied in the real planner, DO NOT overwrite it.
2) If a day/slot is empty, write the preview meal into it.
3) Desserts and Snackitizers:
   - follow the same “only fill empty” rule
   - do not overwrite existing user-added desserts/snackitizers

After confirm:
- Close overlay and return to Planner
- The Planner calendar should now show the new meals
- The “scheduled” state applies (per P1), and midnight auto-count logic works as designed.

IMPORTANT:
- Do NOT automatically mark meals as cooked.
- Do NOT modify consumption totals directly; those update when cooked or auto-counted.
- If your system currently syncs planner -> cart, do not force cart updates in this prompt; leave existing behavior intact.

────────────────────────────────────────

- “Regenerate” rebuilds ONLY the preview overlay plan (not the real calendar).
- Regenerate must respect current checkbox and serving size settings.
- Regenerate should avoid repeating the exact same recipes if alternatives exist.
